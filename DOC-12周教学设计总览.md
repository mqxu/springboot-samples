# 12 周教学设计总览

| 周次 | 主题 | 项目阶段 | 当周核心任务 | 课后作业（含自动化评阅思路） |
| --- | --- | --- | --- | --- |
| 第 1 周 | 大模型编程思想 + Spring Boot 核心认知 | 认知期 | Hello / 系统健康接口 + 初级文档模板 | 学生在 Git 提交 Hello/Health 接口代码与对应规格文档；CI 运行基础单元测试或简单脚本，请求 `/api/hello` / `/api/health`，校验是否返回约定 JSON 结构（包含 `code/msg/data` 等字段）。 |
| 第 2 周 | SDD 基础 + RESTful / Swagger + Postman 测试 | 认知期 | 学生查询接口（单实体查询） | 提交学生查询接口规格文档、Controller/Service 代码以及 Postman Collection；CI 使用 Newman 执行接口测试用例，检查所有用例是否通过（含正常场景与异常场景）。 |
| 第 3 周 | MyBatis-Plus + 数据库设计 | 基础能力期 | 学生信息管理 CRUD | 要求完成学生信息管理的增删改查功能及数据库表设计文档；CI 运行单元测试（覆盖新增、查询、更新、删除），并使用简单 SQL 校验脚本检查关键字段、主键与索引是否按文档设计创建。 |
| 第 4 周 | Service 层设计 + 依赖注入 + 分页 | 基础能力期 | 学生分页查询与业务规则封装 | 完成分页查询接口及相关业务规则文档；CI 通过接口测试脚本检查分页返回结构（`total/list/pageNum/pageSize` 等）是否正确，并验证首页、最后一页、越界页等边界条件行为。 |
| 第 5 周 | 异常处理 + 错误码体系 | 核心技能期 | 图书 / 表白异常处理与错误码表 | 在图书借阅或匿名表白案例中实现统一异常处理和错误码设计，并形成错误码规格文档；单元测试中模拟各类异常场景，CI 中使用 Newman 或 curl 批量请求，验证错误响应的 HTTP 状态码与错误码、错误信息是否与文档一致。 |
| 第 6 周 | 分页优化 + 参数校验 + AI 代码优化 | 核心技能期 | 图书查询 / 表白列表的分页 + 参数校验 + 优化 | 学生对已有列表查询接口增加参数校验与分页优化，并用 AI 优化代码；CI 中使用一组固定请求数据，比较优化前后接口平均响应时间（简单阈值判断），并通过接口测试脚本验证常见参数错误场景是否被正确拦截与返回标准错误信息。 |
| 第 7 周 | JWT 权限控制 + 安全合规 | 核心技能期 | 登录 / 鉴权 / 角色控制 | 完成登录接口、JWT 生成与解析、权限拦截器及相关文档（权限矩阵）；接口测试脚本覆盖未登录访问、Token 错误或过期、普通用户访问管理员接口等场景；CI 中执行这些脚本，检查是否返回预期的错误码与提示信息，并简单校验 Token 过期逻辑。 |
| 第 8 周 | Redis 缓存 + 性能优化 | 核心技能期 | 热门接口缓存（热门图书 / 热门表白） | 学生为热点查询接口设计并实现 Redis 缓存策略（含 key 命名、失效时间），并提交缓存设计文档；使用 JMeter 或简单压测脚本对比开启缓存前后的平均响应时间和 QPS，CI 中检查是否达到设定的性能提升比例，并通过脚本抽样检查 Redis key 命名是否符合规范。 |
| 第 9 周 | AOP 日志 + 监控 + 问题排查 | 实战期 | 操作日志切面 + 问题排查演练 | 实现 AOP 操作日志切面和关键监控点，编写日志规范文档；CI 中通过正则匹配检查日志输出格式是否符合约定，统计错误日志数量；学生需提交至少一份“问题排查报告”，说明如何基于日志与 AI 分析并解决一次故障。 |
| 第 10 周 | 团队协作 + 接口联调（中医平台模块拆分） | 实战期 | 江苏中医知识服务云平台模块划分与接口协同 | 按小组划分江苏中医知识服务云平台后端模块（如用户/资讯/学习/评测/专家答疑/知识库等，参考《中医在线一期》《中医在线二期》PPT），完成模块规格文档、接口清单与 Swagger 配置，每组提交 Swagger JSON 与联调用的 Postman/Newman Collection；CI 对各组核心接口执行联调脚本，检查依赖接口是否可用、返回结构是否符合约定。 |
| 第 11 周 | Docker 容器化部署 + 项目优化 | 实战期 | 江苏中医知识服务云平台后端 Docker 化 + 性能 / 结构优化 | 学生为江苏中医知识服务云平台后端编写 Dockerfile 与（可选）docker-compose 配置，并提交部署说明文档与优化记录；CI 中执行 `docker build` 与 `docker run`，通过探活脚本访问容器内接口，检查容器化部署是否成功，同时可运行一小组性能用例验证优化效果。 |
| 第 12 周 | 综合项目验收 + 经验分享 | 实战期 | 江苏中医知识服务云平台后端综合验收 | 每位学生/小组提交完整项目交付包（规格文档、AI 使用记录、代码、测试报告、部署脚本、复盘总结）；CI 统一跑一次全量自动化测试（单元测试 + 接口测试 + 简单压力测试），教师结合自动化结果与课堂展示、文档质量、答辩表现给出综合评分。 |

---

> 备注：  
> - 上表中的 “CI”“Newman”“JMeter” 等自动化工具可根据学校实际环境调整为 Jenkins、GitLab CI、GitHub Actions 等。  
> - 对于暂不具备完整 CI 条件的班级，可先人工执行部分脚本，逐步过渡到全自动评阅。
